# 模型回退策略（Model Fallback Policy）

## 概述

Vibe Coding Agent 系统支持定义 **主模型（primary）** 与 **回退模型（fallback）**。主模型是默认执行推理的模型；当主模型因特定原因不可用时，系统可自动切换到预定义的回退模型，以保证工作流的连续性。回退模型可在模型配置中声明多个，系统按优先级顺序依次尝试。

## 允许的回退触发条件

仅以下情况可触发模型回退：

| 触发条件 | 说明 |
|---|---|
| `quota_exhausted` | API 配额耗尽 |
| `rate_limited` | 请求频率超限 |
| `provider_unavailable` | 服务商不可用 |
| `timeout_before_mutation` | 在变异操作前超时 |
| `tool_unavailable_before_mutation` | 在变异操作前工具不可用 |

**重要原则**：回退只能在**可能产生变异的操作之前**触发，以防止部分写入或状态不一致。

## 禁止的回退触发条件

以下情况**严禁**触发模型回退：

| 触发条件 | 说明 |
|---|---|
| `acceptance_failed` | 内容验收失败 |
| `review_failed` | 代码审查失败 |
| `scope_violation` | 作用域违规 |
| `security_violation` | 安全违规 |

这些条件代表结果已产生但被判定为不合格。回退不能解决根本问题，因此必须以失败告终，由用户介入处理。

## 审批快照的绑定效应

`approval-snapshot.json` 文件在审批通过时，会记录当前的 `model_policy`（包括主模型与回退配置）。该快照在审批生效后即**冻结**模型策略——此后所有决策必须基于快照中的策略执行，不受后续配置变更的影响。这保证了审批行为的可追溯性与一致性。

## 状态文件与工单文件的数据分离

系统严格区分两类文件：

- **`state.json`**：可变运行时状态。包含当前执行进度、会话上下文、中间结果等动态信息，可在运行中更新。
- **`work-order.json`**：不可变用户输入。一旦创建即视为只读，任何修改都视为新的工单。

这一分离确保了用户意图不会被运行时状态意外污染。

## attempts/fb-* 记录的不可变性

每次回退尝试会生成形如 `attempts/fb-<timestamp>.json` 的追加记录。这些记录具有以下特性：

- **追加写入**：只新增，不修改已有记录。
- **不可变更**：一旦写入，任何工具或用户都不得编辑或删除。
- **审计用途**：用于事后审计、调试与分析。

## 脏工作区策略

当工作区存在未提交的修改（dirty worktree）时，系统在执行回退决策前必须先完成以下步骤之一：

1. 自动暂存（stash）当前修改；
2. 拒绝回退并要求用户先手动处理脏工作区。

默认采用暂存策略。暂存后的回退执行完毕后，系统应尝试恢复暂存内容。恢复失败时，向用户报告并保留暂存记录。

## 推送/合并/部署的独立审批

即使模型回退处于激活状态，**push/merge/deploy** 操作仍然需要独立的人工审批。回退模型自动批准的工作结果不得自动触发推送、合并或部署。这三类操作始终要求人的显式确认，以确保生产安全。
