## жөҒзЁӢжӯҘйӘӨ

### 1. Work Order еүҚзҪ®дёҖж¬Ўе®Ўжү№

- з”ЁжҲ·жҸҗдәӨд»»еҠЎпјҲWork OrderпјүеҗҺпјҢ**дёҚзӣҙжҺҘжү§иЎҢ**гҖӮ
- зі»з»ҹе°Ҷ Work Order е‘ҲзҺ°з»ҷз”ЁжҲ·иҝӣиЎҢ **дёҖж¬Ўе®Ўжү№зЎ®и®Ө**гҖӮ
- з”ЁжҲ·зЎ®и®ӨеҗҺпјҢе·ҘдҪңжөҒжүҚдјҡиҝӣе…Ҙжү§иЎҢйҳ¶ж®өгҖӮ

### 2. implementer иҮӘеҠЁжү§иЎҢ

- е®Ўжү№йҖҡиҝҮеҗҺпјҢ`implementer` agent еңЁ **Debian vibeworker** дёҠиҮӘеҠЁжү§иЎҢд»Јз Ғзј–еҶҷд»»еҠЎгҖӮ
- е®һзҺ°ж №жҚ® Work Order дёӯзҡ„йңҖжұӮе®ҢжҲҗзј–з ҒгҖӮ

### 3. acceptance иҮӘеҠЁйӘҢж”¶

- `implementer` е®ҢжҲҗеҗҺпјҢ`acceptance` agent иҮӘеҠЁиҝҗиЎҢйӘҢж”¶жөӢиҜ•гҖӮ
- йӘҢиҜҒе®һзҺ°жҳҜеҗҰж»Ўи¶і Work Order дёӯе®ҡд№үзҡ„йңҖжұӮгҖӮ
- йӘҢж”¶йҖҡиҝҮжүҚиғҪз»§з»ӯдёӢдёҖжӯҘгҖӮ

### 4. Detached Commit

- йӘҢж”¶йҖҡиҝҮеҗҺпјҢд»Јз Ғд»Ҙ **detached HEAD** зҠ¶жҖҒжҸҗдәӨеҲ°жң¬ең° Git д»“еә“гҖӮ
- дёҚдјҡжҺЁйҖҒеҲ°д»»дҪ•иҝңзЁӢеҲҶж”ҜгҖӮ
- жҸҗдәӨдҝЎжҒҜеҢ…еҗ« Work Order зҡ„жҸҸиҝ°е’Ң job_idгҖӮ

### 5. reviewer иҮӘеҠЁе®ЎжҹҘ

- жҸҗдәӨе®ҢжҲҗеҗҺпјҢ`reviewer` agent иҮӘеҠЁеҜ№д»Јз ҒеҸҳжӣҙиҝӣиЎҢе®ЎжҹҘгҖӮ
- е®ЎжҹҘз»“жһңиҫ“еҮәеҲ°е·ҘдҪңжөҒж—Ҙеҝ—дёӯгҖӮ

### 6. Autonomous Merge (жҺЁиҚҗ)

- **вҡ пёҸ зҰҒжӯўиЈё `gh pr merge`**пјҡжүҖжңү merge еҝ…йЎ»йҖҡиҝҮ `scripts/vibe_autonomous_merge.py`гҖӮ
- Wrapper дјҡиҮӘеҠЁжү§иЎҢ gate йӘҢиҜҒпјҢеҸӘжңү `allow_merge=true` ж—¶жүҚе…Ғи®ё mergeгҖӮ
- дҪҝз”Ёж–№жі•пјҡ

```bash
# Dry-run жЁЎејҸпјҲжҺЁиҚҗе…Ҳжү§иЎҢпјү
python3 scripts/vibe_autonomous_merge.py \
  --repo k176060444-lgtm/vibe-coding-repo \
  --pr <PR_NUMBER> \
  --expected-base-sha <BASE_SHA> \
  --expected-head-sha <HEAD_SHA> \
  --allowed-path <PATH1> \
  --allowed-path <PATH2> \
  --job-id <JOB_ID> \
  --dry-run

# жү§иЎҢ mergeпјҲд»…еҪ“ dry-run йҖҡиҝҮеҗҺпјү
python3 scripts/vibe_autonomous_merge.py \
  --repo k176060444-lgtm/vibe-coding-repo \
  --pr <PR_NUMBER> \
  --expected-base-sha <BASE_SHA> \
  --expected-head-sha <HEAD_SHA> \
  --allowed-path <PATH1> \
  --allowed-path <PATH2> \
  --job-id <JOB_ID>
```

### 7. Post-Merge Freeze

- merge е®ҢжҲҗеҗҺпјҢеҝ…йЎ»жү§иЎҢ post-merge freezeпјҡ
  1. fetch origin е№¶йӘҢиҜҒж–° main SHA
  2. йӘҢиҜҒ merge commit parents
  3. йӘҢиҜҒ changed_paths
  4. еңЁе№ІеҮҖ worktree дёҠиҝҗиЎҢжөӢиҜ•
  5. йӘҢиҜҒ locked job зҠ¶жҖҒдёҚеҸҳ
  6. з”ҹжҲҗ freeze report

## Queue Advisor

Queue Advisor жҸҗдҫӣд»»еҠЎйҳҹеҲ—зҡ„дёӢдёҖжӯҘеҠЁдҪңе»әи®®гҖӮ

**йҮҚиҰҒ**: Queue Advisor v6 е…·еӨҮ superseded job жЈҖжөӢгҖҒnon-production дјҳе…Ҳзә§дҝ®жӯЈе’Ң summary з»ҹи®ЎдёҖиҮҙжҖ§гҖӮ

### Job Lifecycle Policy

Queue Advisor v6+ classifies each job into a lifecycle state:

| Lifecycle | иҜҙжҳҺ | еҸҜжё…зҗҶ |
|-----------|------|--------|
| `tainted_lock` | audit_taintedпјҢеҝ…йЎ»ж°ёд№…дҝқз•ҷ | вқҢ |
| `merged` | result_sha е·Іе…Ҙ mainпјҢе·Іе®ҢжҲҗ | вң… (records еҸҜеҪ’жЎЈ) |
| `superseded` | failed дҪҶиў«еҗҺз»ӯжҲҗеҠҹ job жӣҝд»Ј | вң… |
| `non_production` | smoke/fixture/test/debug/legacy | вң… |
| `active` | pending/in_progress/review_passed | вқҢ |
| `failed` | failed дё”жңӘиў«жӣҝд»Ј | вҡ пёҸ йңҖи°ғжҹҘ |
| `unknown` | зјәе°‘ work-order | вҡ пёҸ йңҖжЈҖжҹҘ |

**е…ій”®и§„еҲҷ**пјҡ
- `wo-code-repo-status-001` ж°ёд№…дҝқз•ҷдёә `tainted_lock`пјҢдёҚеҫ—еҲ йҷӨ recordsпјҢдёҚеҫ—и§ЈйҷӨ lock
- `merged`/`superseded`/`non_production` зҡ„ records еҸҜеңЁеҗҺз»ӯз»ҙжҠӨд»»еҠЎдёӯеҪ’жЎЈпјҲйңҖеҚ•зӢ¬жҺҲжқғпјү
- `failed` зҡ„ job йңҖи°ғжҹҘжҳҜеҗҰд»Қжңүзңҹе®һйҳ»еЎһ

### Superseded Job Detection

v6 дјҡжЈҖжөӢе·Іиў«еҗҺз»ӯжҲҗеҠҹд»»еҠЎжӣҝд»Јзҡ„ failed jobпјҡ

- `wo-doc-workflow-001`пјҲfailedпјүвҶ’ иў« `wo-doc-workflow-002`пјҲreview_passedпјҢе·Іе…Ҙ mainпјүsupersede
- `wo-smoke-001`пјҲfailedпјүвҶ’ иў« `wo-smoke-002`пјҲreview_passedпјүsupersede

Superseded job дёҚеҶҚеҲ—дёә HIGH PRIORITYпјҢиҖҢжҳҜеҪ’е…Ҙ `superseded_jobs` еҲ—иЎЁгҖӮ

### Non-Production Priority Fix

v5 зҡ„дјҳе…Ҳзә§йЎәеәҸдёӯ `failed`пјҲPriority 3пјүе…ҲдәҺ `non_production`пјҲPriority 5пјүпјҢеҜјиҮҙ smoke/fixture/test job зҡ„ failed зҠ¶жҖҒиў«еҪ“дҪң HIGH PRIORITYгҖӮ

v6 дҝ®жӯЈпјҡ`non_production` жЈҖжҹҘеңЁ `failed` д№ӢеүҚпјҲPriority 3пјүпјҢsmoke/fixture/test/debug/legacy job еҚідҪҝ failed д№ҹеҪ’е…Ҙ informational_jobsпјҢдёҚдјҡи§ҰеҸ‘ HIGH PRIORITYгҖӮ

### Summary з»ҹи®ЎдёҖиҮҙжҖ§

v6 з»§жүҝ v5 зҡ„дёҖиҮҙжҖ§дҝқиҜҒ...

### Summary з»ҹи®ЎдёҖиҮҙжҖ§

v5 дҝқиҜҒд»ҘдёӢдёҖиҮҙжҖ§и§„еҲҷпјҡ

- `summary.merged_total` = `len(merged_jobs)`пјҲе§Ӣз»ҲжҲҗз«Ӣпјү
- й»ҳи®ӨпјҲmerged йҡҗи—Ҹпјүпјҡ`merged_jobs` еҲ—иЎЁдёәз©әпјҢ`summary.hidden_merged` = `merged_total`
- `--include-merged`пјҡ`merged_jobs` еҲ—иЎЁе®Ңж•ҙпјҢ`summary.hidden_merged` = 0
- `recovered_jobs` дёӯ `outcome=already_merged` зҡ„ job жҳҜ `merged_jobs` зҡ„**еӯҗйӣҶ**пјҢдёҚйҮҚеӨҚи®Ўж•°
- ж–Үжң¬иҫ“еҮәзҡ„ Merged/Recovered/Unresolved и®Ўж•°дёҺ JSON е®Ңе…ЁдёҖиҮҙ

### Result SHA Recovery

еҪ“ review_passed/clean зҡ„ job зјәе°‘ result_sha ж—¶пјҢжҢүдјҳе…Ҳзә§жҒўеӨҚпјҡ

1. **жң¬ең°ж–Үд»¶**: manifest.json / state.json / approval-snapshot.json / run-record.json / review-record.json
2. **Feature branch**: vibedev/{job_id} жҲ– vibedev/wo-{job_id} зҡ„ HEAD
3. **PR merge parent**: main еҺҶеҸІдёӯеҢ…еҗ« job_id зҡ„ merge commit зҡ„ feature parent

### Result SHA Recovery

еҪ“ review_passed/clean зҡ„ job зјәе°‘ result_sha ж—¶пјҢv4 дјҡжҢүдјҳе…Ҳзә§е°қиҜ•жҒўеӨҚпјҡ

1. **жң¬ең°ж–Үд»¶**: manifest.json / state.json / approval-snapshot.json / run-record.json / review-record.json
2. **Feature branch**: vibedev/{job_id} жҲ– vibedev/wo-{job_id} зҡ„ HEAD
3. **PR merge parent**: main еҺҶеҸІдёӯеҢ…еҗ« job_id зҡ„ merge commit зҡ„ feature parent

жҒўеӨҚжҲҗеҠҹеҗҺпјҡ
- иӢҘ recovered result_sha е·ІеңЁ main вҶ’ ж Үи®°дёә already_mergedпјҲдёҚиҝӣе…Ҙ warningsпјү
- иӢҘдёҚеңЁ main вҶ’ ж Үи®°дёә ready_for_mergeпјҲйҷ„еёҰ result_sha_sourceпјү
- жҒўеӨҚеӨұиҙҘ вҶ’ иҫ“еҮә warning: review_passed but missing result_sha

### Actionability и§„еҲҷ

Queue Advisor v4 зҡ„ ready_for_merge е»әи®®йңҖиҰҒеҗҢж—¶ж»Ўи¶іпјҡ

1. **зңҹе®һ Work Order**: job_id дёҚеҢ№й…Қ smoke/fixture/test/debug/legacy/e2e жЁЎејҸ
2. **е®Ўи®Ўжё…жҙҒ**: audit_status=clean
3. **е·ІйҖҡиҝҮе®ЎжҹҘ**: job_status=review_passed
4. **result_sha еӯҳеңЁ**: еҸҜйӘҢиҜҒд»Јз ҒеҸҳжӣҙпјҲжҲ–е·ІжҒўеӨҚпјү
5. **жңӘеҗҲе…Ҙ main**: result_sha дёҚеңЁ main еҺҶеҸІдёӯ

дёҚж»Ўи¶ідёҠиҝ°жқЎд»¶зҡ„ job дјҡиў«еҪ’е…Ҙпјҡ
- **blocked_jobs**: audit_tainted jobпјҲе§Ӣз»Ҳи®Ўе…Ҙ blocked_totalпјү
- **informational_jobs**: smoke/fixture/test/debug/legacy job
- **warnings**: зјәе°‘ work-order дё”жҒўеӨҚеӨұиҙҘзҡ„ job
- **recovered_jobs**: result_sha е·ІжҒўеӨҚзҡ„ job
- **unresolved_jobs**: result_sha жҒўеӨҚеӨұиҙҘзҡ„ job

### Actionability и§„еҲҷ

Queue Advisor v3 зҡ„ ready_for_merge е»әи®®йңҖиҰҒеҗҢж—¶ж»Ўи¶іпјҡ

1. **зңҹе®һ Work Order**: job_id дёҚеҢ№й…Қ smoke/fixture/test/debug/legacy/e2e жЁЎејҸ
2. **е®Ўи®Ўжё…жҙҒ**: audit_status=clean
3. **е·ІйҖҡиҝҮе®ЎжҹҘ**: job_status=review_passed
4. **result_sha еӯҳеңЁ**: еҸҜйӘҢиҜҒд»Јз ҒеҸҳжӣҙ
5. **жңӘеҗҲе…Ҙ main**: result_sha дёҚеңЁ main еҺҶеҸІдёӯ

дёҚж»Ўи¶ідёҠиҝ°жқЎд»¶зҡ„ job дјҡиў«еҪ’е…Ҙпјҡ
- **blocked_jobs**: audit_tainted jobпјҲе§Ӣз»Ҳи®Ўе…Ҙ blocked_totalпјү
- **informational_jobs**: smoke/fixture/test/debug/legacy job
- **warnings**: зјәе°‘ work-order жҲ– result_sha зҡ„ job

**йҮҚиҰҒ**: blocked_total еҚідҪҝеңЁй»ҳи®Өйҡҗи—Ҹ tainted job ж—¶д№ҹеҸҚжҳ зңҹе®һж•°йҮҸгҖӮ

### дҪҝз”Ёж–№жі•

```bash
# жҹҘзңӢжүҖжңүд»»еҠЎе»әи®®пјҲиҮӘеҠЁжҺ’йҷӨе·ІеҗҲе…Ҙ main зҡ„д»»еҠЎпјү
python3 scripts/vibe_queue_advisor.py

# JSON иҫ“еҮә
python3 scripts/vibe_queue_advisor.py --json

# йҷҗеҲ¶жҳҫзӨәж•°йҮҸ
python3 scripts/vibe_queue_advisor.py --limit 5

# еҢ…еҗ« audit_tainted д»»еҠЎ
python3 scripts/vibe_queue_advisor.py --include-tainted

# жҳҫзӨәе·ІеҗҲе…Ҙ main зҡ„д»»еҠЎ
python3 scripts/vibe_queue_advisor.py --include-merged
```

### Merged-State жЈҖжөӢ

Queue Advisor v2 дјҡжЈҖжҹҘжҜҸдёӘ job зҡ„ `result_sha` жҳҜеҗҰе·ІеңЁ main еҺҶеҸІдёӯпјҡ

## Recommendation Consistency Rule (v2)

### Execution Gate

The execution gate is a machine-checkable admission control before Work Order execution:

```
requirement вҶ’ intake вҶ’ validate вҶ’ registry вҶ’ packager вҶ’ approval-receipt
                                                              вҶ“
                              execution-gate.check вҶҗ [before execute]
                                                              вҶ“
                              verdict: ALLOW / REVIEW / BLOCK
```

Gate checks:
1. Registry status is approved
2. Approval receipt exists and matches
3. Base SHA matches current origin/main
4. Risk level requires human approval if high/critical
5. Stop conditions are defined
6. Allowed paths are specified
7. Forbidden actions include high-risk protections
8. Audit tainted lock is not set

BLOCK verdicts must halt execution. REVIEW verdicts require human decision.

## Fixture / Partial Evidence Handling

When the evidence verifier examines evidence from fixture runs, partial executions,
or level-based unfreeze tests, it uses heuristics to detect **fixture mode**:

### Fixture Mode Detection

The verifier considers evidence to be in fixture mode when:
- `workorder_id` contains `fixture`, `level1`, `level2`, `level3`, or `level4`
- Evidence has `wrapper_dry_run` or `wrapper_merge` set (PR workflow)
- `implementer_model` is `none` or empty (no real model invoked)

### Verdict Classification

| Verdict Detail | Meaning | Operator Action |
|----------------|---------|-----------------|
| `PASS` | All checks passed | Proceed |
| `WARN_EXPECTED_FIXTURE_MODE` | Missing fields expected in fixture mode | Acceptable for fixture runs |
| `WARN_UNEXPECTED_MISSING_FIELD` | Missing fields in non-fixture mode | Investigate вҖ” normally required |
| `FAIL` | Critical integrity failure | Do NOT proceed |

### Operator Summary

The `operator_summary` field provides a human-readable explanation:
- **Fixture mode**: "Fixture/partial evidence mode detected. Missing fields are expected: [list]."
- **Unexpected**: "UNEXPECTED warnings in non-fixture mode: [list]. Investigate why."
- **Fail**: "CRITICAL: Evidence has integrity failures. Errors: [list]."

### Missing Fields

The `missing_fields` list identifies all absent fields across WARN/FAIL checks.
In fixture mode, expected missing fields include: `registry_entry`, `approval_receipt`, `smoke_result`.


## Workflow Quality Gate

The **quality gate** (`scripts/vibe_quality_gate.py`) provides a single-command
aggregated health check for the entire autonomous loop. Run it before and after
every real executor invocation to verify system readiness.

### Usage

```bash
# Full output
python3 scripts/vibe_quality_gate.py

# JSON for automation
python3 scripts/vibe_quality_gate.py --json

# Compact one-liner
python3 scripts/vibe_quality_gate.py --compact

# Via router
python scripts/vibe_command_router.py qg --json
python scripts/vibe_command_router.py go-no-go
```

### Verdict Rules

| Verdict | Meaning | Operator Action |
|---------|---------|-----------------|
| **PASS** | All core checks passed | Proceed with execution |
| **WARN** | Acceptable degradation | Review warnings, then proceed if justified |
| **BLOCK** | Critical failure | Do NOT proceed вҖ” investigate and fix |

### When BLOCK triggers:
- Smoke suite has failures
- origin/main is unreachable or mismatched
- Audit lock (`wo-code-repo-status-001`) is missing or `push_allowed` is not `false`
- Quality gate script itself is missing

### When WARN triggers:
- Loop summary is unavailable
- Evidence verifier is missing
- Router version check failed
- Non-critical component degradation

### Integration with Real Executor

```
Pre-execution:  quality-gate вҶ’ if PASS/WARN вҶ’ proceed
Post-execution: quality-gate вҶ’ verify no regressions
```

## Privileged Push Approval Workflow

High-privilege GitHub keys are controlled via a two-stage approval workflow:

1. **Create approval request**: "priv-approval create --action-id <id> --repo <repo> --branch <branch> --action push --base-sha <sha>"
2. **Human approves**: "approve" / "жү№еҮҶ" / "зЎ®и®Ө" (short approval when exactly 1 pending)
3. **Privileged push (dry-run)**: "priv-push --action-id <id>" вҖ” validates constraints, outputs would_push=true/false
4. **Future**: real push only after dry-run PASS + explicit authorization

### Invariants
- Default: no privileged action is available
- Short approval requires exactly 1 pending, non-expired action
- No force push, no PR merge, no secrets/CI/workflow/provider/SSH paths
- All actions are audit-logged in approval-dir






## V1.5.1 Worker Resilience & Resume

**When worker is temporarily unreachable, do NOT restart the batch. Wait for auto-retry.**

### Recovery States
| State | Meaning | Action |
|-------|---------|--------|
| `WAITING_WORKER_RECOVERY` | Worker unreachable, retrying | Wait, no mutation |
| `RECONCILING` | Worker back, verifying state | Check baseline/worktree/remote |
| `BLOCKED_NEEDS_OPERATOR` | Max wait exceeded | Human intervention needed |

### Retry Policy
- Retry interval: 5 minutes
- Max wait: 75 minutes
- Max retries: 15
- Status report: every 15 minutes

### Resume Rules
- `before_any_mutation` вҶ’ restart WO from scratch
- `after_push` / `after_pr` / `after_merge` вҶ’ reconcile first, then continue/skip/block
- Baseline mismatch or dirty worktree вҶ’ BLOCK, do not auto-fix


## V1.4 Trusted Self Batch Runner

**One-line principle: trusted self repo can batch auto-execute; any blocker stops immediately; external repo writes still require human authorization.**

### Batch Execution Chain
```
batch plan вҶ’ [WO1 вҶ’ WO2 вҶ’ ... вҶ’ WO_N] вҶ’ batch report
  Each WO: branch вҶ’ commit вҶ’ push вҶ’ PR вҶ’ wrapper merge вҶ’ smoke/qg/v1-freeze
  After each WO: refresh baseline before next
  On any failure: STOP, generate batch report, do not continue
```

### Stop Rules
Any of these stops the batch immediately:
- smoke fail, QG fail, V1-freeze fail
- dirty worktree, merge conflict
- forbidden path, token redaction fail
- wrapper merge fail, unexpected changed_paths
- external repo write without approval
